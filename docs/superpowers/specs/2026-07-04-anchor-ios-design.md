## 1. Context and constraints

- Development machine is **Windows 11**: no Xcode, no Swift toolchain, no iOS simulator. PROMPT.md assumes a Mac.
- The user chose (explicit decision, 2026-07-04): **all code is written locally; every verification gate runs on GitHub Actions macOS runners** in a **new public repo `HugoReel/anchor-ios`**. Public means free macOS minutes and unauthenticated run observation (no `gh`/token exists on this machine).
- Everything in PROMPT.md ¬ß3 (design principles), ¬ß4 (stack), ¬ß7 (features), ¬ß12 (hard constraints) is fixed. Open points it delegates: exact package tree, entity field lists, project generation mechanics, verification loop mechanics.

## 2. Project generation ‚ÄĒ decision: XcodeGen

An `.xcodeproj` cannot be authored by Xcode here. Approaches considered:

| Approach | Trade-offs |
|---|---|
| **A. XcodeGen (chosen)** | Declarative `project.yml`, diffable, tiny; generated `.xcodeproj` is gitignored and regenerated by `make project` on any Mac or in CI (`brew install xcodegen`). Build tooling only ‚ÄĒ allowed by PROMPT.md ¬ß4 ("Build tooling (SwiftLint) is fine"). |
| B. Hand-written `project.pbxproj` | No tooling dependency, but the format is undocumented, merge-hostile, and easy to corrupt blind ‚ÄĒ high risk with no local way to validate. |
| C. Tuist | More powerful than needed (caching, graphs); heavier install; its Project.swift manifests require a Swift toolchain to edit confidently. YAGNI. |

The app target stays a **composition root only** (PROMPT.md ¬ß5): `AnchorApp.swift`, dependency container, asset catalog, bundled fonts, Info.plist keys in `project.yml`.

## 3. Module layout ‚ÄĒ decision: one local package, eleven targets

PROMPT.md sketches `Packages/` with eight entries. Two ways to realise it:

- **Eight separate SwiftPM packages** ‚ÄĒ eight manifests, path-based cross-dependencies, awkward all-at-once testing via xcodebuild.
- **One local package `Packages/Anchor` with one target per module (chosen)** ‚ÄĒ a single `Package.swift`; module boundaries and the dependency direction are enforced per-target by SwiftPM exactly as strictly as with separate packages; one auto-generated `Anchor-Package` scheme tests everything in one xcodebuild invocation. The source tree still reads `Sources/AnchorCore`, `Sources/FeatureToday`, ‚Ä¶ as the spec intends.

Targets and allowed dependencies (arrows = "may import"):

```
AnchorCore          ‚Üí (nothing but Foundation)
AnchorDesign        ‚Üí AnchorCore
AnchorPersistence   ‚Üí AnchorCore            (SwiftData lives ONLY here)
AnchorPlatform      ‚Üí AnchorCore            (system adapters: clock, notifications, haptics)
FeatureToday        ‚Üí AnchorCore, AnchorDesign
FeatureTimeline     ‚Üí AnchorCore, AnchorDesign
FeatureGoals        ‚Üí AnchorCore, AnchorDesign
FeatureReflect      ‚Üí AnchorCore, AnchorDesign
FeatureCoping       ‚Üí AnchorCore, AnchorDesign
FeatureSettings     ‚Üí AnchorCore, AnchorDesign
FeatureOnboarding   ‚Üí AnchorCore, AnchorDesign
App target (xcodeproj) ‚Üí all products        (wires repositories/adapters into ViewModels)
```

Notes on deltas from the PROMPT.md sketch, per its "finalise in your plan":
- **FeatureCoping** added: the coping bank must be reachable in ‚Č§2 taps from anywhere (¬ß7.7), so it is a shared feature product, not buried in Settings.
- **FeatureOnboarding** added: keeps the 3-question first run (¬ß7.9) out of the app target, preserving "composition root only".
- **AnchorPlatform** added: `UNUserNotificationCenter`, haptics, and the real clock need `import UserNotifications`/`UIKit`; those imports belong in neither Core (pure) nor Persistence (storage).
- Repository **protocols live in AnchorCore**; SwiftData implementations in AnchorPersistence; `InMemory*` implementations in AnchorCore (used by tests and previews without importing SwiftData).
- Features never import AnchorPersistence or AnchorPlatform ‚ÄĒ they receive protocol values via initializer injection from the app target.

Swift settings: Swift 6 language mode (`swiftLanguageModes: [.v6]`), strict concurrency (default in 6), `platforms: [.iOS(.v17)]`, warnings-as-errors in CI.

## 4. Verification loop ‚ÄĒ CI architecture

Single workflow `.github/workflows/ci.yml`, triggered on every push:

1. **Job `test` (macos-15)** ‚ÄĒ checkout ‚Üí `brew install xcodegen swiftlint` ‚Üí `swiftlint --strict` ‚Üí package tests: `xcodebuild test -scheme Anchor-Package -destination 'platform=iOS Simulator,name=<resolved>'` run from `Packages/Anchor` (xcodebuild auto-generates the package scheme; no project needed) ‚Üí `xcodegen generate` ‚Üí app build: `xcodebuild build -scheme Anchor -destination <sim> CODE_SIGNING_ALLOWED=NO SWIFT_TREAT_WARNINGS_AS_ERRORS=YES`.
2. **Simulator resolution:** a script step picks the first available iPhone from `xcrun simctl list devices available` (preferring "iPhone 16"), so image updates never break the pin.
3. **Observability without auth (critical):** the GitHub API serves run *status* unauthenticated on public repos, but *logs* require a token. So a final `if: always()` step distills the xcodebuild/swiftlint output (error lines, warning lines, test summary, exit codes) into `ci/<branch>/latest.md` and force-pushes it to an orphan **`ci-logs` branch** using the workflow's built-in `GITHUB_TOKEN`. The Windows session then reads both:
   - run conclusion: `GET /repos/HugoReel/anchor-ios/actions/runs?head_sha=‚Ä¶`
   - failure detail: `https://raw.githubusercontent.com/HugoReel/anchor-ios/ci-logs/ci/<branch>/latest.md`
   Full logs additionally upload as an artifact for humans.
4. **Phase gate = green run.** A phase is complete only when its final push is green (build + tests + lint, zero warnings), per PROMPT.md ¬ß2/¬ß9.

## 5. Entity schema v1 (SwiftData, behind repositories)

Conventions: every entity has `id: UUID` (stable), `createdAt: Date`, `modifiedAt: Date`. Enums persist as `String` raw values. Day-scoped queries (`DayPlan.date`, check-ins) are indexed via `#Index`. `VersionedSchema` from v1 with a `SchemaMigrationPlan`, plus one migration test (v1 ‚Üí v1-with-added-optional dry run pattern).

- **DayPlan** ‚ÄĒ `date: Date` (start-of-day, unique per day, indexed), `mode: ScheduleMode` (`clock`|`sequence`), `blocks: [TimeBlock]` (cascade), `notes: String?`
- **TimeBlock** ‚ÄĒ `dayPlan` inverse, `title: String`, `category: BlockCategory` (`focus`, `care`, `home`, `connect`, `out`, `rest` ‚ÄĒ five pastels + Rest as first-class), `startTime: Date?` + `durationMinutes: Int?` (clock mode), `orderIndex: Int` (sequence order and clock display tiebreak), `isFlexible: Bool`, `notes: String?`, `steps: [BlockStep]` (cascade), `state: BlockState` (`notStarted`|`done` ‚ÄĒ **no "missed" state exists in the model**, by design principle 3/5), `completedAt: Date?`
- **BlockStep** ‚ÄĒ `block` inverse, `title: String`, `isDone: Bool`, `orderIndex: Int`
- **DayTemplate** ‚ÄĒ `name: String`, `mode: ScheduleMode`, `blocks: [TemplateBlock]` (cascade)
- **TemplateBlock** ‚ÄĒ `template` inverse, `title`, `category`, `durationMinutes: Int?`, `orderIndex`, `stepTitles: [String]`, `isFlexible: Bool`
- **Goal** ‚ÄĒ `title: String`, `note: String?`, `targetDate: Date?` (never rendered as countdown), `isArchived: Bool`, `steps: [GoalStep]` (cascade)
- **GoalStep** ‚ÄĒ `goal` inverse, `title`, `isDone`, `completedAt: Date?`, `orderIndex`, `ifThenPlans: [IfThenPlan]` (cascade)
- **IfThenPlan** ‚ÄĒ `goalStep` inverse, `triggerKind: TriggerKind` (`time`|`situation`), `situationText: String?`, `triggerTime: Date?` (time-of-day; surfaces on Today within its window), `isActive: Bool`
- **MoodCheckIn** ‚ÄĒ `date: Date` (indexed), `bodySensations: [String]`, `energy: Int?` (1‚Äď5), `valence: Double?`, `arousal: Double?` (‚ąí1‚Ä¶1 sliders, both optional), `emotionWords: [String]`, `isUnsure: Bool` ("I'm not sure" first-class), `note: String?`
- **JournalEntry** ‚ÄĒ `date: Date` (indexed), `text: String` (autosaved, no minimum), `promptShown: String?`
- **EnergyCheckIn** ‚ÄĒ `date: Date` (start-of-day, unique, indexed), `level: Int` (1‚Äď5), `note: String?`
- **WinEvent** ‚ÄĒ `date: Date` (indexed), `kind: WinKind` (`blockDone`, `stepDone`, `goalStepDone`, `checkIn`, `journal`, `rest`, `showedUp`), `label: String?`, `sourceID: UUID?`. Additive only; nothing ever deletes or resets these.
- **CopingStrategy** ‚ÄĒ `title`, `note: String?`, `category: String?`, `orderIndex`, `isSeedExample: Bool`
- **UserPreferences** (single row, fixed UUID) ‚ÄĒ `theme: ThemeChoice` (`calm`|`cool`|`warm`|`lowLight`), `motion: MotionLevel` (`full`|`reduced`|`none`), `hapticsEnabled: Bool` (soft only), `soundEnabled: Bool` (default **false**), `showWins: Bool`, `winsPaused: Bool`, `lowDemandMode: Bool`, `transitionLeadMinutes: Int` (5‚Äď30, default 15), `wakeStartMinutes: Int?`/`wakeEndMinutes: Int?` (minutes from midnight), reflection reminder fields (`remindDaily/Weekly/Monthly/Yearly: Bool` + `‚Ä¶TimeMinutes: Int?` each), `quietStartMinutes: Int?`/`quietEndMinutes: Int?`, `onboardingComplete: Bool`, `seedDataInserted: Bool`

Repositories (protocols in Core, one per aggregate): `DayPlanRepository`, `TemplateRepository`, `GoalRepository`, `ReflectionRepository` (mood + journal), `EnergyRepository`, `WinRepository`, `CopingRepository`, `PreferencesRepository`. Each gets a SwiftData implementation and an InMemory implementation.

`FeatureFlag` enum + `FeatureFlagStore` (UserDefaults-backed) in Core; every ¬ß11 future feature gets a dark flag and a `FUTURE.md` seam entry.

## 6. Testing strategy ‚ÄĒ TDD adapted to a remote verifier

Swift Testing (`@Test`, Xcode 16 toolchain) for unit tests; XCTest only if XCUITests are added in Phase 5/6.

The superpowers TDD loop requires *watching tests fail*. With verification remote, a literal red-run per micro-step would cost dozens of CI round-trips per phase. Adaptation (recorded in DECISIONS.md):

- **Phase 2 (domain core ‚ÄĒ the logic the spec forbids shipping untested):** literal red first. Push the test suite plus compiling stubs that return neutral placeholder values (empty arrays, zero durations) so every test **fails with an assertion message rather than crashing the runner**; confirm the CI run is red with the expected failure messages via ci-logs; then push implementations; confirm green.
- **Later phases:** tests are still written before implementation locally, but each push bundles test+implementation; the green run proves the suite passes, and any test that passes trivially gets a mutation sanity-check during the phase's code review instead of a dedicated red run.
- Coverage gate: ‚Č•80% on AnchorCore, measured by `xcodebuild -enableCodeCoverage YES` + `xccov` summary in the ci-logs report.

## 7. Design skills application (PROMPT.md ¬ß2 items 3‚Äď4, ¬ß8)

Both installed design skills apply to `AnchorDesign` and every screen:

- **frontend-design** (Anthropic): drives distinctive, non-templated direction ‚ÄĒ typography personality, deliberate palette, signature element per screen.
- **impeccable**: setup will run against this repo in Phase 1/3 (`PRODUCT.md` + `DESIGN.md` created via its init flow, product register); its general rules (contrast ‚Č•4.5:1 verified, no side-stripe borders, no gradient text, motion easing/reduced-motion rules, z-index scale) become part of the review checklist for every screen even though its automated detector targets web files, not SwiftUI.

**Standing conflict rule (pre-declared):** where either skill's push for boldness/aesthetic risk conflicts with PROMPT.md ¬ß3 principles (calm, predictable, low-arousal, mature), **the spec's principles win** ‚ÄĒ distinctiveness is expressed through typography, spacing rhythm, and the anchor motif instead of saturated color or motion. Each concrete conflict still gets a one-line note in DECISIONS.md as the spec requires.

## 8. Process adaptations (recorded honestly)

- This session is autonomous; the user approved the requirements by handing over PROMPT.md ("follow it end-to-end") and answered the two blocking environment questions. Section-by-section design re-approval is therefore not re-requested; instead this spec + the plan are committed early so the user can redirect at any point.
- superpowers' worktree isolation is satisfied by working in a fresh dedicated repo on `main` with the user's knowledge (there is no shared workspace to protect; PROMPT.md itself instructs committing to an empty repo).
- Subagent-driven execution is used selectively: this environment cannot run any build locally, so implementer subagents cannot self-verify; the CI round-trip is the verifier. Subagents are reserved for parallel-safe authoring (e.g. independent feature screens in Phase 3) and for the final whole-branch review, per PROMPT.md ¬ß2.2.

## 9. Risks

| Risk | Mitigation |
|---|---|
| Blind Swift: first pushes may fail to compile | Conservative, spec-checked Swift; small early pushes to calibrate; ci-logs gives exact compiler errors |
| macOS runner image drift (Xcode/simulator names) | Dynamic simulator resolution; Xcode version left at image default, recorded in ci-logs |
| Actions queue/latency | Batch work per push; poll with backoff |
| SwiftData API subtleties (relationships, `#Index`, migrations) unverifiable locally | Schema kept simple; migration test from day one; repository protocol seam means schema fixes don't ripple |
| Lexend download/licensing | Fetch official release from google/fonts repo including OFL.txt; commit both |
| User never creates the GitHub repo | All work commits locally; push task stays open and is the only blocked item |
